## 📋 โครงสร้างข้อมูล (sheet `line OA`)

ข้อมูลทั้งหมดอยู่ใน sheet เดียว 12 คอลัมน์ ใช้คอลัมน์ `type` แยกประเภท (`faq` / `product` / `license`):

| คอลัมน์ | หน้าที่ |
|---|---|
| `id` | รหัสแถว |
| `type` | ประเภท: faq / product / license |
| `category` | หมวด |
| `keywords` | คำค้น/คำพ้องที่ลูกค้าพิมพ์ (ใช้ match คำถาม) |
| `answer` | คำตอบ/ข้อมูลหลัก |
| `url` | ลิงก์อ้างอิง |
| `brand` | ยี่ห้อ (เฉพาะ product) |
| `model_code` | ชื่อรุ่น (เฉพาะ product) |
| `price` | ราคาขายจริง (เฉพาะ product) |
| `price_pack` | ราคาแพ็ค/โปร (เฉพาะ product) |
| `license_required` | ต้องขอ / ไม่ต้องขอ (เฉพาะ product) |
| `license_ref` | id แถว license ที่เกี่ยวข้อง (เฉพาะ product) |

> (แก้บั๊กแล้ว 2026-07-07) เดิมเอกสารนี้ยังมีคอลัมน์ `stock` (in/out) อยู่ ทั้งที่ชีตจริงและ `lib/sheet.ts` เอาคอลัมน์นี้ออกไปแล้วตั้งแต่ก่อนหน้านี้ ไม่มี logic เช็ค stock เหลืออยู่ในโค้ดเลย — ลบออกจากเอกสารให้ตรงกับของจริงแล้ว

**หลักการค้น:** ค้นจากคอลัมน์ `keywords` ก่อนเสมอ เจอแถวไหนดูที่ `type` แล้วตอบตามแบบของประเภทนั้น

## เรื่องที่ 1 — ตอบราคา/ข้อมูลสินค้า

เมื่อลูกค้าถามราคา/ข้อมูลรุ่นไหน → match `keywords` เจอแถว `type=product` → ตอบตาม template:

```
{brand} {model_code} ราคา {price} บาท ค่ะ 📻
• {answer}
• {license_text} รับประกัน 2 ปี
ดูรายละเอียด/ราคาล่าสุด: {url}
ลูกค้าสนใจสั่งเลยไหมคะ?
```

**ตัวอย่างจริง (TC-5M):**
```
Spender TC-5M ราคา 3,560 บาท ค่ะ 📻
• วิทยุใส่ซิม POC Network Radio ใช้งานทั่วไป
• ไม่ต้องขอใบอนุญาต ใช้ได้ถูกกฎหมาย รับประกัน 2 ปี
ดูรายละเอียด/ราคาล่าสุด: https://spenderclub.com/product/spender-tc-5m/
ลูกค้าสนใจสั่งเลยไหมคะ?
```

**กฎแปลง `license_required` → `license_text`:**
- `ไม่ต้องขอ` → "ไม่ต้องขอใบอนุญาต ใช้ได้ถูกกฎหมาย"
- `ต้องขอ` → "ถูกกฎหมาย (รุ่นนี้ต้องทำใบอนุญาต สอบถามรายละเอียดได้ค่ะ)"

**กรณีพิเศษ (ต้องจัดการ ไม่งั้นตอบพลาด):**

*1.1 ราคาเป็น `-` (ขายแค่แพ็ค):* ใช้ `price_pack` แทน
```
{brand} {model_code} — {price_pack} ค่ะ 📻
```
ตัวอย่าง: "Spender UBC-410T — แพ็คคู่ 2 เครื่อง 2,790 ค่ะ"

*1.2 มี `price_pack` เพิ่ม (มีทั้งเดี่ยวและแพ็ค):* ต่อท้าย "หรือ {price_pack} (โปรโมชั่นสุดคุ้ม)"

*1.3 กำกวม (match หลายรุ่น เช่น "245"):* อย่าเดา ให้เสนอตัวเลือก
```
รบกวนยืนยันรุ่นที่สนใจค่ะ มีหลายรุ่นใกล้เคียง:
1) {model_1}  2) {model_2}  3) {model_3}
แจ้งชื่อรุ่นได้เลยค่ะ
```

## เรื่องที่ 2 — ตอบใบอนุญาตเฉพาะรุ่น (เชื่อม 2 แถว)

เมื่อลูกค้าถาม "รุ่นนี้ต้องขอ/ต้องทำใบอนุญาตไหม" → ทำ 2 ขั้น:
1. อ่าน `license_required` จากแถวสินค้า → บอกต้องขอ/ไม่ต้องขอ
2. ตามคอลัมน์ `license_ref` → ไปดึง `answer` + `url` จากแถว `type=license` ที่ `id` ตรงกัน → ดึงค่าธรรมเนียม + ขั้นตอนมาตอบรวมกัน

**ห้ามตอบแค่ "ต้องขอค่ะ"** — ต้องครบ: ต้องขอไหม + ค่าธรรมเนียม + ขั้นตอน + ทางลัดถามแอดมิน

**กรณี "ต้องขอ":**
```
{brand} {model_code} รุ่นนี้ต้องทำใบอนุญาตค่ะ 📋
เป็น{category}
• ค่าธรรมเนียม {fee จากแถว license_ref}
• ดูขั้นตอนการขอใบอนุญาตได้ที่นี่ {url จากแถว license_ref}
  หรือสอบถามแอดมินช่วยดำเนินการได้ค่ะ
```
**ตัวอย่างจริง (TC-246H Plus → license_ref = lic-245-5w):**
```
Spender TC-246H Plus รุ่นนี้ต้องทำใบอนุญาตค่ะ 📋
เป็นวิทยุ 245MHz กำลังส่ง 5W (เครื่องมีทะเบียน)
• ค่าธรรมเนียม 535 บาท/เครื่อง
• ดูขั้นตอนการขอใบอนุญาตได้ที่นี่ https://rep.tccom.co.th/
  หรือสอบถามแอดมินช่วยดำเนินการได้ค่ะ
```

**กรณี "ไม่ต้องขอ":**
```
{brand} {model_code} ไม่ต้องขอใบอนุญาตค่ะ ใช้งานได้ถูกกฎหมายเลย 😊
เป็น{category} {answer}
```

> **กลไกเชื่อม:** คอลัมน์ `license_ref` ในแถวสินค้าชี้ไป id แถว license แล้ว
> (TC-246H→lic-245-5w, ราชการ→lic-gov, สมัครเล่น→lic-amateur, ใส่ซิม→lic-simradio)
> แค่ lookup ตาม id ไม่ต้องเดาประเภทเอง

## เรื่องที่ 3 — ค้นเว็บเมื่อไม่เจอในชีต (ก่อน handoff)

ถ้าค้นในชีตไม่เจอ **อย่าเพิ่งโยนแอดมิน** ให้ค้นเว็บก่อน:
```
ค้นชีต (keywords) ไม่เจอ
   → ค้น site:spenderclub.com และ site:spendernetwork.com
   → เจอ: ตอบ + แนบ url
   → ไม่เจอ/ไม่ชัด: handoff แอดมิน + log คำถาม
```

**Template เมื่อค้นเว็บเจอ:**
```
{ข้อมูลที่ได้จากเว็บ} ค่ะ 📻
ดูรายละเอียดเพิ่มเติม: {url ที่เจอบนเว็บ}
หากต้องการข้อมูลเพิ่มเติม สามารถสอบถามเพิ่มเติมได้เลยนะคะ
```

**Template เมื่อค้นเว็บแล้วยังไม่เจอ:**
```
ขออภัยค่ะ ยังไม่พบข้อมูลที่ตรงกับคำถาม 🙏
รบกวนแจ้งรายละเอียดเพิ่มเติม หรือรอแอดมินติดต่อกลับในเวลาทำการ 08:00–17:00 น. นะคะ
```

**กฎสำคัญ:**
1. **อ่านราคาให้ถูก** — ราคา**ขีดฆ่า** = ก่อนลด, ราคา**ไม่ขีดฆ่า** = ราคาขายจริง → ใช้ราคาขายจริงตอบ
2. **ห้ามแต่งข้อมูล** — เว็บไม่ชัด ห้ามเดา → handoff แทน
3. **กัน loop/ค้นซ้ำถี่** — cooldown ต่อ user 2 นาที + cache ผลค้นเว็บ 24 ชม.
4. **บันทึกคำถามที่ตอบไม่ได้** — เก็บ log ให้ Marketing เอาไปเติมชีต
5. **Auto-cache (ทำทีหลังได้ ไม่ต้องรีบ)** — ผลค้นเว็บเขียนกลับเข้าชีตแถวใหม่ ช่วงแรกแค่ค้นเว็บแล้วตอบก็พอ
6. **ห้ามเดาจากสินค้าใกล้เคียง** — ถ้าผลค้นหาไม่ตรงกับสิ่งที่ลูกค้าถามโดยตรง:
   - ห้ามหยิบสินค้ารุ่นอื่น/ใกล้เคียงมาตอบแทน
   - ห้ามแนบลิงก์ที่ไม่ตรงคำถาม
   - ห้ามสรุปสถานะ (หมด/มีของ/ราคา) ที่ผลค้นไม่ได้ระบุชัดว่าตรงกับสินค้านั้น
   - ให้ตอบว่าไม่พบข้อมูล + handoff อย่างเดียว

> URL pattern: สินค้า = `spenderclub.com/product/{brand}-{model}/`  
> URL ที่ส่งให้ลูกค้า: ตัด tracking parameters ออกก่อนเสมอ (เช่น `?srsltid=...`, `?utm_source=...`)

## เรื่องที่ 4 — แก้ flow นอกเวลาทำการ (18:00–07:59)

**ปัญหาเดิม:** ข้อความแรกของวันนอกเวลา บอทส่ง notice แล้วหยุด ไม่ตอบคำถาม
→ ลูกค้าถามราคาตอนกลางคืนแล้วโดนเงียบ ต้องพิมพ์ซ้ำ เสี่ยงเสียลูกค้า

**แก้เป็น:** prepend notice ไว้หน้าคำตอบ แล้วตอบคำถามต่อในข้อความเดียวกัน
(ใช้รูปแบบเดียวกับ Greeting Card ในเวลาทำการที่ prepend แล้วไปต่อ)

```
ข้อความแรกของวัน (นอกเวลา):
  off_hours:{userId} ไม่มี key → set key (23h)
  → prepend notice หน้าคำตอบ → ตอบคำถามต่อ (ข้อความเดียว)

ข้อความถัดไปในคืนเดียวกัน:
  off_hours:{userId} มีแล้ว → ไม่ prepend → ตอบปกติ
```

**notice ที่ใช้:**
```
ขณะนี้อยู่นอกเวลาทำการ แอดมินจะตอบกลับในเวลาทำการ 08:00–17:00 น. ค่ะ 🙏
ระหว่างนี้น้องใจดีช่วยดูแลก่อนนะคะ
```

> handoffMsg นอกเวลา + greeting card คงเดิม ไม่ต้องแก้
> **ทดสอบ:** ทักครั้งแรก 3 ทุ่ม "TC-246H ราคาเท่าไหร่" → ต้องได้ notice + ราคา 4,150 ในข้อความเดียว

## เรื่องที่ 5 — ปรับ flow อ่านภาพ (อุดช่องโหว่)

flow เดิมใช้ได้ แต่เพิ่ม 3 จุดนี้ให้ดีขึ้น:

**5.1 ตอบรับทันทีเมื่อได้รับรูป** (อย่าเงียบ 5 นาที)
เมื่อลูกค้าส่งรูป → ตอบรับสั้นๆ ทันที แล้วค่อยรอดูว่าจะพิมพ์ถามต่อไหม
```
ได้รับรูปแล้วค่ะ กำลังดูให้นะคะ 📷
```
(logic รอ 5 นาที + 4-ปุ่ม card คงเดิม แค่เพิ่มข้อความตอบรับนำหน้า)

**5.2 กรณี OCR อ่านรูปไม่ออก / ไม่ใช่รูปสินค้า**
ถ้า OCR ไม่เจอชื่อรุ่น → อย่าแสดง card เปล่า ให้ถามตรงๆ:
```
รบกวนพิมพ์ชื่อรุ่นที่สนใจได้ไหมคะ จะได้ช่วยหาข้อมูลให้ถูกต้องค่ะ
```

**5.3 4-ปุ่ม card ต้องผูกกับรุ่นที่ OCR เจอ**
เมื่อลูกค้ากด [ราคา]/[สเปค] บอทต้องรู้ว่าถามรุ่นไหน โดยดึง model_code
จากชื่อสินค้าที่เพิ่งบันทึกลง history ตอน OCR — ไม่ใช่ถามลอยๆ

## เรื่องที่ 6 — ปรับ flow IMEI (อุดช่องโหว่)

flow 4 ขั้นเดิมใช้ได้ แต่เพิ่ม 2 จุดนี้:

**6.1 ให้ยกเลิกกลางคันได้**
ระหว่างอยู่ใน imei_protocol ถ้าลูกค้าพิมพ์ "ยกเลิก" หรือถามเรื่องอื่นชัดเจน
→ ออกจาก protocol ล้าง state ที่ค้าง ไม่วนถามซ้ำ
```
ยกเลิกรายการแล้วค่ะ มีอะไรให้น้องใจดีช่วยอีกไหมคะ
```

**6.2 กัน "ใช่" ที่กำกวมในขั้นยืนยัน (สำคัญ)**
ขั้น 4 ถ้าลูกค้าตอบ "ใช่" พร้อมข้อมูลเพิ่ม/เปลี่ยน เช่น "ใช่ แต่เปลี่ยนเป็นกลุ่ม C"
หรือ "ใช่ แล้วเพิ่มอีกเครื่อง" → **อย่า handoff ทันที**
ให้จับว่ามีข้อมูลใหม่ แล้วกลับไปขั้น 3 สรุปยืนยันใหม่
handoff เฉพาะเมื่อลูกค้ายืนยันชัดเจนไม่มีเงื่อนไขเพิ่ม
→ กันแอดมินได้ข้อมูลผิด

> ขั้นตอน 1-4 เดิม (ตรวจ 15 หลัก, ถามประเภท, สรุป, HANDOFF:) คงไว้ทั้งหมด

## เรื่องที่ 7 — กันถาม/ตอบซ้ำ + spam (3 ชั้น)

**ปัญหา:** ลูกค้าถามเดิมซ้ำๆ บอทตอบเป๊ะเดิมทุกรอบ (ดูไม่ฉลาด) และมีคนพิมพ์เล่น/ยิงรัว

**หลักการสำคัญ:** ตอบครั้งแรกเต็มเสมอ แล้วค่อยปรับเมื่อซ้ำ — อย่าเงียบใส่ลูกค้าตั้งแต่แรก
(บางคนถามซ้ำเพราะเน็ตหลุด/ไม่เห็นคำตอบ ไม่ใช่กวน)

**ชั้นที่ 1 — กันถามซ้ำเป๊ะ (last_question)**
จำ**คำถามล่าสุด**ที่ลูกค้าพิมพ์ (ไม่ใช่คำตอบบอท) เช็ค**ก่อน**เรียก Gemini เลย ถ้าคำถามใหม่ตรงกับคำถามก่อนหน้าเป๊ะภายใน 2 นาที
→ ข้าม Gemini ไปเลย เปลี่ยนเป็นข้อความสั้นแทนการตอบยาวซ้ำ:
```
ราคา TC-246H อยู่ที่ 4,150 บาทตามที่แจ้งไปนะคะ มีอะไรให้ช่วยเพิ่มไหมคะ 😊
```

> (แก้บั๊กแล้ว 2026-07-11) เดิมเช็คหลังเรียก Gemini เสร็จ โดยเทียบ "คำตอบที่ได้" กับคำตอบก่อนหน้า (key `last_answer`) — Gemini ตั้ง `temperature: 1.0` ทำให้คำตอบไม่เหมือนเดิมทุกตัวอักษรได้แม้ถามคำถามคลุมเครือซ้ำเป๊ะ (เช่น "สอบถามเพิ่มเติม") เลยหลุดจากการเช็ค ตอบเต็มรูปแบบซ้ำไปเรื่อยๆ ไม่มีวันเงียบ ตอนนี้ย้ายมาเช็คก่อนเรียก Gemini แทน เทียบ "คำถามลูกค้า" (ค่าที่แน่นอน ไม่มีความสุ่ม) แทนคำตอบ — เปลี่ยน key เป็น `last_question` ทุกช่องทาง (LINE/Facebook ทั้ง text และ postback/Web Chat) แม่นยำกว่าเดิมและประหยัด token เพิ่มด้วยเพราะข้ามการเรียก Gemini ไปเลยถ้าตรวจพบว่าซ้ำ — พร้อมกันนี้แก้เพิ่มให้ Facebook postback (ปุ่มบนการ์ดสินค้า เช่น "สอบถามเพิ่มเติม") มีระบบกันซ้ำนี้ด้วย เดิมไม่มีเลยเพราะเป็นคนละ code path จาก text flow

**ชั้นที่ 2 — กัน spam ยิงรัว (rate limit)**
ถ้า user ส่งเกิน 5 ข้อความใน 10 วินาที → ชะลอ ตอบรวบครั้งเดียว หรือส่ง:
```
รอสักครู่นะคะ น้องใจดีกำลังอ่านข้อความค่ะ 🙏
```

**ชั้นที่ 3 — กันพิมพ์เล่น/ไร้สาระ (nonsense)**
ข้อความที่ไม่ใช่คำถาม (เช่น "5555", ".", สุ่มตัวอักษร) ซ้ำๆ
→ ตอบสั้นครั้งแรก แล้วเงียบถ้ายังกวนต่อ (ใช้ logic เดียวกับ out_of_domain เดิม)

**Redis keys เพิ่ม:**
| Key | TTL | หน้าที่ |
|---|---|---|
| `last_question:{userId}` | 2 นาที | จำคำถามล่าสุด กันถามซ้ำเป๊ะ (เช็คก่อนเรียก Gemini) |
| `msg_rate:{userId}` | 10 วินาที | นับความถี่ข้อความ กัน spam |
| `nonsense_count:{userId}` | 10 นาที | นับข้อความไร้สาระติดกัน |

> **จุดห้ามแตะ:** อย่าให้กลไกนี้บล็อกลูกค้าจริง — ตอบครั้งแรกเต็มเสมอ
> ใช้ rate limit/nonsense เฉพาะเมื่อมีรูปแบบซ้ำชัดเจน ไม่ใช่เงียบใส่ทุกข้อความสั้น

## เรื่องที่ 8 — Exact Keyword Match (ทางลัดตอบจากชีต ไม่ผ่าน Gemini)

> เพิ่มหลัง INSTRUCTIONS.md ฉบับแรก — ไฟล์: `lib/sheet.ts` (`findExactMatch()`) ใช้เหมือนกันทั้ง 3 ช่องทาง เช็คก่อนเรียก Gemini เสมอ

**เงื่อนไข (ต้องผ่านทั้ง 2 ชั้น):**
1. ข้อความ (lowercase, trim) ยาวไม่เกิน 40 ตัวอักษร และมี keyword จากคอลัมน์ `keywords` (คั่นด้วย comma/slash/pipe) เป็น substring โดยข้อความยาวไม่เกิน keyword ที่ match + 15 ตัวอักษร
2. ข้อความต้องไม่มีคำที่บ่งบอกความต้องการอื่น: `เสีย|ซ่อม|พัง|ปัญหา|เคลม|ต่างกัน|เปรียบเทียบ|ดีกว่า|ไม่เอา|ไม่ใช่|ยกเลิก|แนะนำ|งบ|เหมาะกับ`

**ผลลัพธ์:** match ได้พอดี 1 แถว → ตอบทันทีด้วย template ธรรมชาติ (`customerProduct()`/`customerFaq()`/`customerLicense()` — คนละชุดกับ `formatProduct()`/`formatFaq()`/`formatLicense()` ที่เป็นบริบทดิบให้ Gemini อ่าน) **ไม่เรียก Gemini เลย** · match 0 หรือ ≥2 แถว → ปล่อยผ่านไป Gemini ตามปกติ

## เรื่องที่ 11 — เคลม/ส่งซ่อม: เช็ค FAQ ก่อน + handoff แทนการวินิจฉัยเอง (ทุกช่องทาง)

> เพิ่มหลัง INSTRUCTIONS.md ฉบับแรก — ไฟล์: `lib/prompts.ts` (`<repair_protocol>`, `buildSystemPrompt(..., channel)`), `app/api/web-chat/route.ts` (`CLAIM_RE`, `CLAIM_MSG`)
> ปรับปรุงล่าสุด 2026-07-10 (รอบ 2) — เดิมทำเฉพาะ Web Chat (2026-07-07) → ปรับเป็น logic เดียวกันทุกช่องทาง (2026-07-10 รอบ 1) → ปรับ Web Chat เพิ่มอีกรอบให้ข้ามขั้นถามข้อมูลไปเลย (รอบ 2)

**ขั้นที่ 1 — เช็ค `<faq>` ก่อนเสมอ (เหมือนกันทุกช่องทาง):** ถ้าเป็นคำถามขั้นตอน/นโยบายทั่วไป (ไม่ใช่แจ้งอาการเสียของเครื่องตัวเองโดยเฉพาะ) เช่น "จะส่งเครื่องเคลม" "เช็คสถานะซ่อม" "รับประกันกี่ปี" → ตอบจาก `<faq>` ทันที ไม่ต้องถามข้อมูลเพิ่ม (ในชีตมีคำตอบพร้อมอยู่แล้ว เช่น `warranty-repair-02`, `warranty-repair-04`)

**ขั้นที่ 2 — ถ้าแจ้งอาการเฉพาะเจาะจง (ไม่มีคำตอบตรงใน `<faq>`) — ต่างกันตาม channel:**

| ช่องทาง | บอททำ | ทำไมถึงต่าง |
|---|---|---|
| LINE / Facebook | ถามข้อมูล 3 อย่างก่อนเสมอ (รุ่น/อาการ/ซื้อมานานแค่ไหน) แล้วตอบ `HANDOFF: ซ่อม/เคลม \| รุ่น: [รุ่น] \| อาการ: [อาการ] \| ซื้อมานาน: [ระยะเวลา]` → `notifyAdmin()`/`notifyAdminFacebook()` ส่งสรุปเข้ากลุ่มแอดมินจริง + ตอบ handoffMsg | มี real-time admin คอยรับสรุป คุ้มที่จะถามเก็บบริบทไว้ |
| Web Chat | **ข้ามขั้นถามไปเลย** ตอบ `HANDOFF: ซ่อม/เคลม` ทันทีคำเดียว (ไม่มีรุ่น/อาการ/ระยะเวลาต่อท้าย) → ตรวจจับแล้วตอบ `CLAIM_MSG` | ไม่มี real-time admin ให้ส่งสรุปไปหา ถ้าถามแล้วข้อมูลไม่ถูกใช้ที่ไหนเลย เสียเวลาลูกค้าเปล่าๆ แล้วต้องอธิบายซ้ำเองตอนติดต่อแอดมินจริงอยู่ดี |

**ห้ามวิเคราะห์อาการหรือแนะนำแก้เองอีกแล้วในทุกช่องทาง** — ส่งต่อให้คนช่วยดูแลแทนเพื่อความถูกต้องแม่นยำที่สุดเสมอ

**Web Chat มีชั้นกรองเพิ่มอีกชั้นก่อนถึง Gemini:** ข้อความที่มีคำว่า "เคลม"/"ส่งซ่อม"/"claim" ตรงตัว (เช่น "จะส่งเครื่องเคลมครับ") จะถูก `CLAIM_RE` ดักไว้ ตอบทันทีไม่ผ่าน Gemini เลย:
```
เรื่องเคลม/ส่งซ่อมรบกวนติดต่อแอดมินโดยตรงเลยนะคะ
แอดไลน์ @spenderclub
เบอร์โทรติดต่อ: 085-222-9111, 089-661-2682
จันทร์–เสาร์ เวลา 08:00–17:00 น.ค่ะ
```
ข้อความที่แจ้งอาการโดยไม่มีคำเหล่านี้ (เช่น "PTT กดไม่ติดครับ") จะไม่ถูกดักตรงนี้ ไหลไปเรียก Gemini ตามขั้นที่ 1-2 ด้านบนแทน — ถ้าจบที่ `HANDOFF: ซ่อม/เคลม` ก็จะได้ข้อความเดียวกันนี้อยู่ดี (คนละเส้นทาง ผลลัพธ์เดียวกัน)

> (แก้บั๊กแล้ว 2026-07-10) เดิม `<repair_protocol>` เข้าทันทีเมื่อเจอคำว่า "เครื่องเสีย/ซ่อม/เคลม" แบบไม่มีเงื่อนไข ก่อนขั้นค้น FAQ เสมอ ทำให้คำถามทั่วไปที่ชีตตอบได้อยู่แล้วถูกถามข้อมูลซ้ำโดยไม่จำเป็น และเดิมมีขั้นตอนให้ Gemini วิเคราะห์อาการ/แนะนำแก้เอง/แนะนำที่อยู่ส่งซ่อมด้วยตัวเอง — ตอนนี้เปลี่ยนเป็นเช็ค FAQ ก่อนเสมอ และให้คนช่วยดูแลประเมินอาการเฉพาะเจาะจงแทน Gemini ทั้งหมด (Web Chat ยังปรับเพิ่มอีกรอบให้ข้ามขั้นถามข้อมูลไปเลย เพราะข้อมูลที่เก็บมาไม่มีที่ให้ใช้ต่อ — ไม่มี `notifyAdmin()` สำหรับ Web Chat)

## เรื่องที่ 12 — Facebook: ปุ่มการ์ดสินค้า + กันตอบซ้ำใน postback

> เพิ่มหลัง INSTRUCTIONS.md ฉบับแรก (2026-07-11) — ไฟล์: `app/api/fb-webhook/route.ts` (`fbSendProductCard()`, postback handler)

**1. เปลี่ยนปุ่ม "ติดต่อแอดมิน" → "สอบถามเพิ่มเติม" บนการ์ดสินค้า**

การ์ดสินค้า (Generic Template ที่ส่งเมื่อคำตอบ Gemini มี URL `spenderclub.com`) มี 2 ปุ่ม: "ดูรายละเอียด/ราคา" (web_url) และปุ่มที่ 2 เดิมชื่อ "ติดต่อแอดมิน" (postback) — กดแล้วเข้า `shouldHandoff()` ตอบ handoffMsg ทันที **โดยไม่เคยเรียก `notifyAdminFacebook()` เลย** ลูกค้าเห็นข้อความ "รอแอดมินติดต่อกลับนะคะ" แต่ไม่มีใครในทีมรู้เรื่องจริง

แก้เป็น **"สอบถามเพิ่มเติม"** แทน (payload ยังเป็น `CONTACT_ADMIN` เหมือนเดิม — ไม่กระทบเพราะไม่ตรงกับ 5 payload ที่โค้ดรู้จักอยู่แล้ว) คำนี้ไม่ชนคำใน `shouldHandoff()` trigger list เลย จึงไหลไปเรียก Gemini จริง ได้คำตอบขยายความจากบริบทสินค้าที่เพิ่งแนะนำไป (conversation history เก็บคำตอบเต็มของ Gemini ไว้อยู่แล้วแม้ลูกค้าจะเห็นแค่การ์ดแบบย่อ)

**2. เพิ่มระบบกันตอบซ้ำใน postback handler**

พบว่าปุ่ม postback ทั้งหมด (ไม่ใช่แค่ "สอบถามเพิ่มเติม") ไม่มีระบบกันตอบซ้ำเลย ต่างจาก flow ข้อความพิมพ์ปกติที่มีอยู่แล้ว (เรื่องที่ 7 ชั้น 1) — กดปุ่มเดิมซ้ำกี่ครั้งก็เรียก Gemini ใหม่ทุกครั้ง ได้คำตอบเต็มซ้ำไปเรื่อยๆ ไม่มีวันเงียบ (ยืนยันจากการทดสอบจริง กดซ้ำ 3 ครั้งได้คำตอบเดิมเป๊ะทุกครั้ง)

ตอนนี้เพิ่ม logic กันซ้ำชุดเดียวกับ text flow เข้าไปในนี้ด้วยแล้ว (ใช้ Redis key เดียวกัน `last_question:{userId}`/`repeat_count:{userId}` เทียบ `queryText` ก่อนเรียก Gemini เลย — ครั้งที่ 2 ที่ซ้ำ: reminder สั้นๆ · ครั้งที่ 3+: เงียบ) ครอบคลุมทุกปุ่ม postback ไม่ใช่แค่ปุ่มที่เพิ่งแก้ (ดูเรื่องที่ 7 สำหรับรายละเอียดว่าทำไมถึงเปลี่ยนจากเทียบ `last_answer` เป็น `last_question`)

> **ทำไม LINE ไม่มีปัญหานี้:** ปุ่มใน LINE Flex Message ใช้ `action: { type: 'message' }` — กดแล้วส่งเป็นข้อความพิมพ์ปกติกลับมา ไหลผ่าน text handler เดียวกับข้อความทั่วไปที่มีระบบกันตอบซ้ำอยู่แล้วโดยอัตโนมัติ ไม่ต้องแก้อะไรเพิ่ม (Web Chat ไม่มีปุ่มแบบนี้เลยไม่เกี่ยวด้วย)

- ❌ แก้ไขหรือย่อ URL ในข้อมูล — คัดลอกมาตรงๆ เท่านั้น
- ❌ ตอบประโยคเดิมซ้ำ — ต้องปรับคำพูดทุกครั้ง

### ขั้นตอนการคิดก่อนตอบ

1. **ขอบคุณ/รับทราบ** ("ขอบคุณ", "ok", "ครับ", "👍") → ตอบ "ยินดีให้บริการค่ะ" แล้วหยุด
2. **ไม่เกี่ยวกับวิทยุสื่อสาร** (อากาศ การเมือง ทำอาหาร) → แจ้งว่าช่วยได้เฉพาะวิทยุสื่อสาร
3. **ชื่อรุ่น หรือตัวเลข ≤10 หลัก** → ค้น FAQ ราคา/ข้อมูลทันที
4. **ตัวเลข ≥11 หลัก** (IMEI) → แจ้งให้พิมพ์ "ขอเพิ่มกลุ่ม" พร้อม IMEI
5. **แนะนำทั่วไป** ("รุ่นไหนดี") → ถามประเภทงานและงบประมาณก่อน
6. **เครื่องเสีย/ซ่อม** → ทำตาม repair protocol
7. **ค้น FAQ** → ถ้าเจอ ตอบทันที
8. **ไม่เจอใน FAQ** → ค้นเว็บ spenderclub.com ก่อนเสมอ
9. **ออกนอกขอบเขต** → ส่งต่อแอดมิน

### Template ตอบราคา/ข้อมูลสินค้า

```
{brand} {model_code} {ราคา} ค่ะ 📻
• {ข้อมูลสินค้า}
• {ใบอนุญาต} รับประกัน 2 ปี
ดูรายละเอียด/ราคาล่าสุด: {URL}
ลูกค้าสนใจสั่งเลยไหมคะ?
```

**กรณีพิเศษ:**
- ราคา = `-` → ใช้ราคาแพ็คแทน
- สินค้าหมด → แจ้งของหมด พร้อม URL ดูรุ่นใกล้เคียง
- match หลายรุ่น → ถามยืนยันรุ่นก่อน

### Template ใบอนุญาต

**ต้องขอใบอนุญาต:**
```
{brand} {model_code} รุ่นนี้ต้องทำใบอนุญาตค่ะ 📋
เป็น{หมวด}
• {ค่าธรรมเนียมและรายละเอียด}
• ดูขั้นตอน: {URL}
  หรือสอบถามแอดมินช่วยดำเนินการได้ค่ะ
```

**ไม่ต้องขอ:**
```
{brand} {model_code} ไม่ต้องขอใบอนุญาตค่ะ ใช้งานได้ถูกกฎหมายเลย 😊
```

### Protocol ซ่อมเครื่อง / เคลม (2 ขั้น — อัปเดต 2026-07-10)

**ขั้น 1** — เช็ค `<faq>` ก่อนเสมอ (เหมือนกันทุกช่องทาง): คำถามขั้นตอน/นโยบายทั่วไป (เช่น "จะส่งเครื่องเคลม", "เช็คสถานะซ่อม", "รับประกันกี่ปี") → ตอบจาก FAQ ทันที ไม่ถามอะไรเพิ่ม

**ขั้น 2** — แจ้งอาการเฉพาะเจาะจง (ไม่มีคำตอบตรงใน FAQ) — ต่างกันตามช่องทาง:
- **LINE/Facebook** — ถามรุ่น, อาการ, ซื้อมานานแค่ไหนก่อน แล้วตอบ `HANDOFF: ซ่อม/เคลม | รุ่น: ... | อาการ: ... | ซื้อมานาน: ...` → แจ้งกลุ่มแอดมิน + ตอบ handoffMsg ปกติ
- **Web Chat** — **ข้ามขั้นถามไปเลย** ตอบ `HANDOFF: ซ่อม/เคลม` ทันทีคำเดียว → ตอบ "ติดต่อแอดมินโดยตรง" แทน (ไม่มี real-time admin ให้ส่งสรุปไปหา ถามแล้วข้อมูลก็ไม่ถูกใช้ต่อ)

**ห้ามวิเคราะห์อาการหรือแนะนำแก้เองอีกแล้วในทุกช่องทาง** — ให้คนช่วยดูแลประเมินแทนเสมอ

### Protocol IMEI (4 ขั้น)

**ขั้น 1** — ตรวจสอบ IMEI ต้องครบ 15 หลัก  
**ขั้น 2** — ถามประเภทคำสั่ง (เพิ่ม/ลบ/ย้าย/เปลี่ยนกลุ่ม)  
**ขั้น 3** — แสดงสรุปให้ยืนยัน  
**ขั้น 4** — ลูกค้ากด "ใช่" เฉยๆ → ส่งต่อแอดมิน  

> ถ้าลูกค้าพิมพ์ "ยกเลิก" ระหว่างทาง → ยกเลิกทันที  
> ถ้า "ใช่ แต่..." หรือมีข้อมูลเพิ่ม → กลับขั้น 3 สรุปใหม่

### ส่งต่อแอดมิน (trigger words)

บอทส่งต่อแอดมินเมื่อลูกค้าพูดถึง — แยก 2 แบบตามว่าต้องค้นชีต/เว็บก่อนไหม (LINE/Facebook เท่านั้น ดู `lib/handoff.ts`):

**ทันที (ข้ามค้นชีต/เว็บ):**
| หัวข้อ | คำที่ trigger |
|---|---|
| ขอคุยกับคน | คุยกับคน, ขอแอดมิน, ติดต่อแอดมิน |
| ร้องเรียน | ฟ้อง, ร้องเรียน, ไม่พอใจ |

**รอก่อน (ค้นชีต/เว็บก่อนเสมอ ไม่เจอจริงถึง handoff):**
| หัวข้อ | คำที่ trigger |
|---|---|
| ขายส่ง | ขายส่ง, wholesale, อยากซื้อจำนวนมาก |
| ใบเสนอราคา | ขอใบเสนอราคา, quotation, ขอ quote |
| ตัวแทน | ตัวแทนจำหน่าย, dealer, franchise, ขออนุญาต |
| สื่อ/PR | ติดต่อสื่อ, PR, interview |
| บริการหลังการขาย (IMEI/กลุ่ม) | ติดต่อศูนย์บริการ, เพิ่มกลุ่ม, ขอเพิ่มกลุ่ม, เพิ่มเครื่อง, ลงทะเบียนกลุ่ม, เพิ่มอุปกรณ์, ขอเพิ่มเครื่อง |

**"ขอเจ้าของ" — พฤติกรรมพิเศษแยกจากกลุ่มอื่นทั้งหมด (`isOwnerRequest()` ใน `lib/handoff.ts`):**
| ช่วงเวลา | บอททำ |
|---|---|
| ในเวลาทำการ (08:00–17:59) | immediate handoff เหมือนกลุ่ม "ทันที" ด้านบน |
| นอกเวลาทำการ (18:00–07:59) | **ไม่ handoff เลย** — ตอบ "ขณะนี้อยู่นอกเวลาทำการค่ะ..." แทนแล้วจบ ไม่แจ้งแอดมิน |

> (แก้บั๊กแล้ว 2026-07-07) เรื่องที่แก้พร้อมกัน:
> 1. เดิม prompt ที่ส่งให้ Gemini (`<out_of_scope_triggers>` ใน `lib/prompts.ts`) สั่งให้ตอบ handoffMsg ทันทีเมื่อเจอ "ขายส่ง" หรือ "ใบเสนอราคา" เหมือนกลุ่ม "ทันที" ด้านบน ทำให้ Gemini ข้ามค้นชีตไปเลยทั้งที่มีคำตอบอยู่จริง (เช่น `documents-01` ตอบเรื่องขอใบเสนอราคา) ตอนนี้เอา 2 กลุ่มนี้ออกจาก `out_of_scope_triggers` แล้ว ให้ค้นชีต/เว็บก่อนเสมอตามตารางด้านบน — แก้เฉพาะ LINE/Facebook (ใช้ `lib/prompts.ts` ร่วมกัน) ส่วน Web Chat ไม่กระทบเพราะมี `shouldHandoff()` กันไว้ก่อนเรียก Gemini อยู่แล้ว (ดูหัวข้อ Anti-Spam ด้านล่าง — Web Chat ไม่แยกทันที/รอก่อน ใช้ list เดียวกันหมด)
> 2. เอกสารเดิมจัด "ขอเจ้าของ" ไว้ในกลุ่ม "ทันที" ปนกับคุยกับคน/ขอแอดมิน/ติดต่อแอดมิน ทั้งที่โค้ดจริงแยกเป็น `isOwnerRequest()` ต่างหาก มีพฤติกรรมตามเวลาที่ไม่เหมือนกลุ่มอื่น (นอกเวลาทำการไม่ handoff เลย) ตอนนี้แยกเป็นตารางของตัวเองแล้ว
> 3. `lib/prompts.ts` มีคำว่า "ขออนุญาต" อยู่ใน `<out_of_scope_triggers>` แต่ `DEFERRED_HANDOFF_TRIGGERS` ใน `lib/handoff.ts` ไม่มีคำนี้เลย ทำให้ 2 ไฟล์ไม่ตรงกัน (Gemini เห็นคำนี้แล้วตอบ handoffMsg แต่โค้ดฝั่ง `shouldHandoffDeferred()` กลับไม่รู้จักคำนี้ ถ้า Gemini คืน NOT_FOUND ข้อความนี้จะตกไปที่ retry flow ปกติแทนที่จะเป็น deferred flow) ตอนนี้เพิ่ม "ขออนุญาต" เข้า `DEFERRED_HANDOFF_TRIGGERS` ให้ตรงกับ `lib/prompts.ts` แล้ว
> 4. เพิ่มแถว "บริการหลังการขาย (IMEI/กลุ่ม)" ในตารางด้านบน — คำกลุ่มนี้อยู่ใน `DEFERRED_HANDOFF_TRIGGERS` มาตั้งแต่ก่อนหน้านี้แล้ว แต่ไม่เคยถูกเขียนไว้ในตารางนี้เลย

### Anti-Spam (Web Chat เท่านั้น)

กฎป้องกัน spam ใช้เฉพาะ **Web Chat Widget** — LINE OA และ Facebook ไม่ได้รับผลกระทบ

| กฎ | เงื่อนไข | ผลลัพธ์ |
|---|---|---|
| Honeypot | field `_hp` ถูกกรอก (bot อัตโนมัติ) | เงียบทันที ไม่ตอบ |
| IP Rate Limit | > 20 ข้อความต่อ 5 นาที (ต่อ IP) | "ขออภัยค่ะ ส่งข้อความถี่เกินไปนิดนึง รบกวนรอ 5 นาที..." |
| Nonsense Filter | ข้อความ < 2 ตัวอักษร หรือซ้ำเดิมทั้งหมด (???, 555, aaa) | "มีอะไรให้น้องใจดีช่วยไหมคะ 😊" ไม่ส่ง Gemini |
| Session Block | out_of_domain ≥ 5 ครั้งใน session เดียว | บล็อก session 30 นาที เงียบหลังจากนั้น |

**หลักการ:** ตอบครั้งแรกเต็มเสมอ บล็อกเมื่อเกิน threshold เท่านั้น

**Redis Keys เพิ่มเติม:**
- `web_ip_rate:{ip}` — TTL 5 นาที
- `web_blocked:{sessionId}` — TTL 30 นาที
