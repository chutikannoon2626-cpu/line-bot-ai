📱 LINE Webhook — Flow ทั้งหมด

ไฟล์: app/api/line-webhook/route.ts · ทุก event process พร้อมกันด้วย Promise.all(body.events.map(...))

เตรียมข้อมูลก่อน TEXT/IMAGE (ทุก event type)

A
Verify LINE Signature
HMAC-SHA256 · header X-Line-Signature · ล้มเหลว → 401 return ทันที
B
Off-Hours Check (ก่อน reply ทุกอัน)
Bangkok time ≥18:00 หรือ <08:00 → GET off_hours:{userId} · ไม่มี = SET EX 23h + เก็บ offHoursNotice text ไว้ prepend ใน reply
C
Greeting Check
shouldGreet(userId) → SET greeted:{userId} NX EX 86400 · return non-null = ทักครั้งแรก (greetFirst=true)

TEXT Message — ลำดับ step ทั้งหมด

1
Log + Load State
logChat({userId, 'LINE', 'user', message}) (fire-and-forget) · getHistory(userId) · getHandoffMessage()
2
Admin Release: คืนบอท:{targetId}
DEL takeover:{targetId} → safeReply "✅ คืนน้องใจดีดูแลลูกค้าแล้วค่ะ ({id.slice(-6)})" → return
3
Schedule Check
isScheduledOff('line') → ถ้า off = เงียบ (return โดยไม่ reply)
4
Rate Limit
INCR msg_rate:{userId} + EXPIRE 10s (pipeline) · เกิน 5: ครั้งแรก reply "รอสักครู่" ครั้งต่อไปเงียบ → return
5
Pending Image Check (img_data:{userId})
GET {id, ts} · Upstash คืนค่าเป็น object ที่ deserialize มาแล้วอัตโนมัติ — เช็ค typeof raw === 'string' ก่อนค่อย JSON.parse (ถ้าเป็น object อยู่แล้วใช้ตรงๆ) เพื่อกันบั๊ก parse ซ้อนที่ทำให้ imgData เป็น null เสมอ · elapsed < 300000ms:
DEL key → ดึงรูป blob → generateReplyWithImage(b64, faq, userMessage) timeout 15s → ans → saveHistory → return
elapsed ≥ 300000ms:
DEL key → OCR ยี่ห้อ/รุ่น (max 50 tokens) → ไม่เจอ: "รบกวนพิมพ์ชื่อรุ่น" / เจอ: imageIntentCard(product) → return
6
Special Command: สอบถามสเปก
GET+DEL img:{userId} · ถ้ามี imageId: ดึงรูป → generateReplyWithImage(b64, faq, 'สอบถามสเปกและฟังก์ชันการใช้งาน') timeout 15s → return
7
Admin Takeover Suppression
GET takeover:{userId} · ถ้ามี = เงียบ (บอทเงียบ แอดมินดูแลอยู่) → return
8
Pre-Handoff State
GETDEL pre_handoff:{userId} · ถ้ามี pending trigger:
SET routed:{userId} EX 300 + SET takeover:{userId} EX 7200 + DEL handoff_notified:{userId} (pipeline) → notifyAdmin() → ans(handoffMsg) → saveHistory → return
9a
"ขอเจ้าของ" นอกเวลาทำการ (isOwnerRequest)
Bangkok thaiHour ≥18 หรือ <8 → ans(OWNER_REQUEST_OFF_HOURS_MSG "ขณะนี้อยู่นอกเวลาทำการค่ะ...") → saveHistory → return
ไม่ handoff เลย ไม่เข้า pre_handoff/routed ใดๆ
9b
Handoff Trigger — ทันที (shouldHandoffImmediate || isOwnerRequest ในเวลาทำการ)
"คุยกับคน/ขอแอดมิน/ติดต่อแอดมิน" (ขอคนโดยตรง) · "ฟ้อง/ร้องเรียน/ไม่พอใจ" (ร้องเรียน) · หรือ "ขอเจ้าของ" เฉพาะในเวลาทำการ 08:00–17:59 — ข้ามค้นชีต/เว็บได้เลย →
already routed (GET routed:{userId}): INCR handoff_notified:{userId} EX 10min · count=1: reply "แอดมินจะติดต่อกลับ" · count≥2: เงียบ → return
ยังไม่ routed: SET pre_handoff:{userId} EX 600 → ans("กรุณาแจ้งรายละเอียด...") → saveHistory → return
9b2
Exact Keyword Match (findExactMatch) — ไม่ผ่าน Gemini
ตรวจ keyword ตรงชีตแบบ deterministic (2 ชั้น: สั้น+ตรง / ไม่มีคำบ่งบอกความต้องการอื่น) — match เจอ 1 แถวพอดี → ans(คำตอบจาก customerProduct/customerFaq/customerLicense) → saveHistory → return
ไม่ match (ไม่เจอ/กำกวม/มีคำอื่นปน) → ไปต่อขั้นถัดไปตามปกติ · ดูรายละเอียดที่หัวข้อ "Exact Keyword Match"
9c
Order Confirm (ORDER_CONFIRM_RE) — ไม่ผ่าน Gemini
"สั่งเลย/อยากสั่ง/ต้องการซื้อ/จะซื้อ/สั่งได้ไหม/ซื้อยังไง/โอนเงิน/ชำระเงิน/จ่ายเงิน" →
นอกเวลาทำการ (≥18 หรือ <8): ans("ขณะนี้อยู่นอกเวลาทำการ โดยเจ้าหน้าที่จะเริ่มให้บริการ...") → saveHistory → return ทุกครั้ง ไม่กันซ้ำ
ในเวลาทำการ: GET order_confirm:{userId} · ไม่มี → SET EX 1200 (20min) → ans("กรุณารอสักครู่นะคะ") → saveHistory → return
มีแล้ว (ยังไม่ครบ 20 นาที) → เงียบ ไม่ตอบซ้ำ → return
9d
กันถามซ้ำเป๊ะ — เช็คก่อนเรียก Gemini
GET last_question:{userId} · ตรงกับ userMessage เป๊ะ → INCR repeat_count:{userId} EX 120s · count=1: ans(reminder) → return · count≥2: เงียบ → return (ไม่เรียก Gemini เลยทั้ง 2 กรณี — ประหยัด token ด้วย)
ไม่ตรง → ไปต่อขั้นถัดไปตามปกติ
(แก้บั๊กแล้ว 2026-07-11) เดิมเช็คหลังเรียก Gemini เสร็จ โดยเทียบ "คำตอบ" ที่ได้กับคำตอบก่อนหน้า — Gemini ตั้ง temperature: 1.0 ทำให้คำตอบไม่เหมือนเดิมทุกตัวอักษรได้แม้ถามคำถามคลุมเครือซ้ำเป๊ะ (เช่น "สอบถามเพิ่มเติม") เลยหลุดจากการเช็ค ตอบเต็มรูปแบบซ้ำไปเรื่อยๆ ไม่มีวันเงียบ — ตอนนี้ย้ายมาเช็คก่อนเรียก Gemini แทน โดยเทียบ "คำถามลูกค้า" (ค่าที่แน่นอน ไม่มีความสุ่ม) แทนคำตอบ แม่นยำกว่าและประหยัด token เพราะข้ามการเรียก Gemini ไปเลยถ้าซ้ำ — เปลี่ยน key จาก last_answer เป็น last_question ทุกช่องทาง (LINE/FB text/FB postback/Web Chat)
10
🤖 Gemini Reply
Promise.race([generateReply(msg, faq, history, handoffMsg, 'line'), timeout(10s)]) · timeout → GEMINI_UNAVAILABLE
คำอื่นที่เคยเป็น handoff trigger (ขายส่ง/ใบเสนอราคา/ตัวแทนจำหน่าย/เพิ่มกลุ่ม ฯลฯ) จะไม่ถูกดักตรงนี้แล้ว — ปล่อยให้ Gemini ค้น FAQ + search_spender_specs ตามปกติก่อนเสมอ
(แก้บั๊กแล้ว 2026-07-07) เดิม <out_of_scope_triggers> ใน lib/prompts.ts สั่ง Gemini ให้ตอบ handoffMsg ทันทีเมื่อเจอคำว่า "ขายส่ง/wholesale/อยากซื้อจำนวนมาก" หรือ "ขอใบเสนอราคา/quotation/ขอ quote" — ทำให้ Gemini ข้ามขั้นตอนค้น FAQ ไปเลยทั้งที่ในชีตมีคำตอบอยู่จริง (เช่น documents-01 ตอบเรื่องขอใบเสนอราคา, order-payment-shipping-03 ตอบเรื่องขายส่ง) ตอนนี้เอา 2 กลุ่มคำนี้ออกจาก out_of_scope_triggers แล้ว ให้ Gemini ค้น FAQ/เว็บก่อนเสมอ ถ้าไม่เจอจริง (NOT_FOUND) ค่อยเข้า shouldHandoffDeferred flow ตามปกติ (ขั้น 11 ด้านล่าง) · แก้ที่ lib/prompts.ts ไฟล์เดียว ใช้ร่วมกับ Web Chat ด้วย แต่ Web Chat ไม่กระทบเพราะมี shouldHandoff() กันไว้ก่อนเรียก Gemini อยู่แล้ว (ดูขั้น 9 ของ Web Chat)
(แก้บั๊กแล้ว 2026-07-10) <repair_protocol> เดิมเข้าทันทีเมื่อเจอคำว่า "เครื่องเสีย/ซ่อม/เคลม" แบบไม่มีเงื่อนไข ก่อนขั้นค้น FAQ เสมอ ทำให้คำถามทั่วไปที่ชีตตอบได้อยู่แล้ว (เช่น "จะส่งเครื่องเคลม" → warranty-repair-02, "เช็คสถานะซ่อม" → warranty-repair-04) ถูกถามข้อมูลซ้ำโดยไม่จำเป็น ตอนนี้แก้ให้เช็ค <faq> ก่อนเสมอสำหรับคำถามขั้นตอน/นโยบายทั่วไป (ขั้นที่ 1 — เหมือนกันทุกช่องทาง) ถ้าเจอตอบจาก FAQ ทันที · ส่วนอาการเสียเฉพาะเจาะจงที่ไม่มีใน FAQ (ขั้นที่ 2 — buildSystemPrompt(..., channel) แยกเนื้อหาตาม channel):
LINE/Facebook: เก็บรุ่น/อาการ/ระยะเวลาซื้อก่อน แล้วตอบ HANDOFF: ซ่อม/เคลม | รุ่น: ... | อาการ: ... | ซื้อมานาน: ... — มี notifyAdmin() ส่งสรุปนี้เข้ากลุ่มแอดมินจริง คุ้มที่จะถาม
Web Chat: ข้ามการถามไปเลย ตอบ HANDOFF: ซ่อม/เคลม ทันทีคำเดียว (2026-07-10 แก้เพิ่ม) — เพราะ Web Chat ไม่มี notifyAdmin() ข้อมูลที่เก็บมาจะไม่ถูกส่งต่อให้ใครเลย เสียเวลาลูกค้าเปล่าๆ แล้วต้องอธิบายซ้ำเองตอนติดต่อแอดมินจริงอยู่ดี
ไม่มีช่องทางไหนวิเคราะห์อาการหรือแนะนำแก้เองอีกแล้ว (ให้แอดมินประเมินแทนเพื่อความแม่นยำ) — ดูผลลัพธ์ HANDOFF ที่ขั้น 11 ด้านล่าง และขั้น 12 ของ Web Chat
11
Handle Gemini Returns (ลำดับ)
GEMINI_UNAVAILABLE → ans(UNAVAILABLE_MSG "ระบบนานกว่าปกติ...") → return
CANCEL_IMEI → ans("ยกเลิกรายการแล้ว") + saveHistory([]) → return
HANDOFF / HANDOFF:xxx → SET routed+takeover + DEL notified → notifyAdmin(summary) → ans(handoffMsg) → return (รวมเคส HANDOFF: ซ่อม/เคลม | ... จาก repair_protocol ด้วย — ใช้ logic เดียวกัน แค่ summary ต่างกัน)
[OUT_OF_DOMAIN] → INCR nonsense_count:{userId} EX 600 · ≤1: ans(OUT_OF_DOMAIN_MSG) · ≥2: เงียบ → return
[NOT_FOUND] → ค้นชีต+เว็บไม่เจอจริง ตรวจ shouldHandoffDeferred ก่อนเสมอ:
มี deferred trigger (ขายส่ง/ใบเสนอราคา/ตัวแทนจำหน่าย/เพิ่มกลุ่ม ฯลฯ): SET pre_handoff:{userId} EX 600 → ans("กรุณาแจ้งรายละเอียด...") → saveHistory → return (ข้าม retry 2 รอบ)
ไม่มี trigger: GET retry:{userId}: ไม่มี = SET EX 600 + ans(retryMsg) + lpush unanswered_log / มี = DEL + ans(handoffMsg) → return
12
Send Reply — ans()
greetFirst=true: safeReply([offHours?, WELCOME_MSG]) → safePush([answer]) · ส่งแยก 2 ข้อความ
greetFirst=false: safeReply([offHours?, answer])
ต่อจากนั้น: SET last_question:{userId} EX 120s (เก็บ userMessage ไว้เทียบซ้ำรอบหน้า — ดูขั้น 9d) + DEL repeat_count · saveHistory · log · zincrby question_freq

📘 Facebook Messenger — Flow

ไฟล์: app/api/fb-webhook/route.ts · userId namespace = fb:{psid} · ไม่มี takeover key (LINE เท่านั้น)

เตรียมข้อมูลก่อน TEXT/IMAGE (ทุก event)

A
Verify FB Signature (X-Hub-Signature-256)
HMAC-SHA256 · crypto.timingSafeEqual() · ล้มเหลว → 401
B
Off-Hours Check + Greeting Check
Bangkok ≥18/<08 → GET off_hours:fb:{psid} · ไม่มี = SET EX 23h + offHoursNotice=true
shouldGreet('fb:{psid}') → greetFirst

TEXT — ลำดับ step

1
logChat(user) + getHistory + Rate Limit
INCR msg_rate:fb:{psid} EX 10s (pipeline) · เกิน 5: ครั้งแรก reply ครั้งต่อไปเงียบ
2
Schedule Check
isScheduledOff('fb') → เงียบ
3
Special: สเปค/ฟังก์ชัน
GET+DEL fb_img_url:fb:{psid} · ถ้ามี: fbSend(offHours?) → ดาวน์โหลดรูปจาก URL → generateReplyWithImage timeout 20s → fbSendReply → return
4
Off-Hours + Greeting ส่งก่อน reply
offHoursNotice=true → fbSend(OFF_HOURS_NOTICE) · greetFirst=true → fbSend(WELCOME_MSG)
5
Pre-Handoff State + Stale Key Check
GETDEL pre_handoff:fb:{psid} · ถ้ามี: ตรวจ history ว่า bot เคยถาม "กรุณาแจ้งรายละเอียด" ไหม (กัน stale) → ผ่าน: SET routed:fb:{psid} EX 300 + DEL notified → notifyAdminFacebook(psid, ...) → fbSend(handoffMsg) → return
5b
"ขอเจ้าของ" นอกเวลาทำการ (isOwnerRequest)
Bangkok thaiHour ≥18 หรือ <8 → fbSend(OWNER_REQUEST_OFF_HOURS_MSG "ขณะนี้อยู่นอกเวลาทำการค่ะ...") → saveHistory → return · ไม่ handoff เลย
6
shouldHandoffImmediate || isOwnerRequest ในเวลาทำการ
"คุยกับคน/ขอแอดมิน/ติดต่อแอดมิน" · "ฟ้อง/ร้องเรียน/ไม่พอใจ" · หรือ "ขอเจ้าของ" เฉพาะเวลาทำการ 08:00–17:59 — ข้ามค้นชีต/เว็บได้เลย
already routed → INCR handoff_notified:fb:{psid} EX 10min · count=1: reply / count≥2: เงียบ
ไม่ routed → SET pre_handoff:fb:{psid} EX 600 → fbSend("กรุณาแจ้งรายละเอียด...") → return
5c
Exact Keyword Match (findExactMatch) — ไม่ผ่าน Gemini
ตรวจ keyword ตรงชีตแบบ deterministic (2 ชั้น) — match เจอ 1 แถวพอดี → fbSend(คำตอบ) → saveHistory → return · ไม่ match → ไปต่อตามปกติ · ดูหัวข้อ "Exact Keyword Match"
6b
Order Confirm (ORDER_CONFIRM_RE) — ไม่ผ่าน Gemini
"สั่งเลย/อยากสั่ง/ต้องการซื้อ/จะซื้อ/สั่งได้ไหม/ซื้อยังไง/โอนเงิน/ชำระเงิน/จ่ายเงิน"
นอกเวลาทำการ → fbSend("ขณะนี้อยู่นอกเวลาทำการ...") ทุกครั้ง ไม่กันซ้ำ
ในเวลาทำการ → GET order_confirm:fb:{psid}: ไม่มี → SET EX 1200 + fbSend("กรุณารอสักครู่นะคะ") · มีแล้ว → เงียบ
7
🤖 Gemini timeout 10s · GEMINI_UNAVAILABLE → UNAVAILABLE_MSG
ใช้ lib/prompts.ts เดียวกับ LINE — (แก้บั๊กแล้ว 2026-07-07) "ขายส่ง/ใบเสนอราคา" เอาออกจาก out_of_scope_triggers แล้ว Gemini ค้น FAQ ก่อนเสมอ (ดูรายละเอียดที่หัวข้อ LINE ขั้น 10)
