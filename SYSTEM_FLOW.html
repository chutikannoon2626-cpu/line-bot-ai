OPTIONS/api/web-chat — CORS preflight · 204
POST/api/web-chat — รับ {sessionId, message, _hp} · CORS whitelist origin

DELbody:{target:'unanswered'|'freq'} — ล้างข้อมูล

GET/api/admin/chatlog?key=&userId= — messages ของ user คนนั้น

GET/api/admin/schedule?key=&channel=line|fb|web

📱 LINE Webhook — Flow ทั้งหมด

ไฟล์: app/api/line-webhook/route.ts · ทุก event process พร้อมกันด้วย Promise.all(body.events.map(...))

เตรียมข้อมูลก่อน TEXT/IMAGE (ทุก event type)

A
Verify LINE Signature
HMAC-SHA256 · header X-Line-Signature · ล้มเหลว → 401 return ทันที
B
Off-Hours Check (ก่อน reply ทุกอัน)
Bangkok time ≥18:00 หรือ <08:00 → GET off_hours:{userId} · ไม่มี = SET EX 23h + เก็บ offHoursNotice text ไว้ prepend ใน reply
C
Greeting Check
shouldGreet(userId) → SET greeted:{userId} NX EX 86400 · return non-null = ทักครั้งแรก (greetFirst=true)

TEXT Message — ลำดับ step ทั้งหมด

1
Log + Load State
logChat({userId, 'LINE', 'user', message}) (fire-and-forget) · getHistory(userId) · getHandoffMessage()
2
Admin Release: คืนบอท:{targetId}
DEL takeover:{targetId} → safeReply "✅ คืนน้องใจดีดูแลลูกค้าแล้วค่ะ ({id.slice(-6)})" → return
3
Schedule Check
isScheduledOff('line') → ถ้า off = เงียบ (return โดยไม่ reply)
4
Rate Limit
INCR msg_rate:{userId} + EXPIRE 10s (pipeline) · เกิน 5: ครั้งแรก reply "รอสักครู่" ครั้งต่อไปเงียบ → return
5
Pending Image Check (img_data:{userId})
GET {id, ts} · Upstash คืนค่าเป็น object ที่ deserialize มาแล้วอัตโนมัติ — เช็ค typeof raw === 'string' ก่อนค่อย JSON.parse (ถ้าเป็น object อยู่แล้วใช้ตรงๆ) เพื่อกันบั๊ก parse ซ้อนที่ทำให้ imgData เป็น null เสมอ · elapsed < 300000ms:
DEL key → ดึงรูป blob → generateReplyWithImage(b64, faq, userMessage) timeout 15s → ans → saveHistory → return
elapsed ≥ 300000ms:
DEL key → OCR ยี่ห้อ/รุ่น (max 50 tokens) → ไม่เจอ: "รบกวนพิมพ์ชื่อรุ่น" / เจอ: imageIntentCard(product) → return
6
Special Command: สอบถามสเปก
GET+DEL img:{userId} · ถ้ามี imageId: ดึงรูป → generateReplyWithImage(b64, faq, 'สอบถามสเปกและฟังก์ชันการใช้งาน') timeout 15s → return
7
Admin Takeover Suppression
GET takeover:{userId} · ถ้ามี = เงียบ (บอทเงียบ แอดมินดูแลอยู่) → return
8
Pre-Handoff State
GETDEL pre_handoff:{userId} · ถ้ามี pending trigger:
SET routed:{userId} EX 300 + SET takeover:{userId} EX 7200 + DEL handoff_notified:{userId} (pipeline) → notifyAdmin() → ans(handoffMsg) → saveHistory → return
9a
"ขอเจ้าของ" นอกเวลาทำการ (isOwnerRequest)
Bangkok thaiHour ≥18 หรือ <8 → ans(OWNER_REQUEST_OFF_HOURS_MSG "ขณะนี้อยู่นอกเวลาทำการค่ะ...") → saveHistory → return
ไม่ handoff เลย ไม่เข้า pre_handoff/routed ใดๆ
9b
Handoff Trigger — ทันที (shouldHandoffImmediate || isOwnerRequest ในเวลาทำการ)
"คุยกับคน/ขอแอดมิน/ติดต่อแอดมิน" (ขอคนโดยตรง) · "ฟ้อง/ร้องเรียน/ไม่พอใจ" (ร้องเรียน) · หรือ "ขอเจ้าของ" เฉพาะในเวลาทำการ 08:00–17:59 — ข้ามค้นชีต/เว็บได้เลย →
already routed (GET routed:{userId}): INCR handoff_notified:{userId} EX 10min · count=1: reply "แอดมินจะติดต่อกลับ" · count≥2: เงียบ → return
ยังไม่ routed: SET pre_handoff:{userId} EX 600 → ans("กรุณาแจ้งรายละเอียด...") → saveHistory → return
10
🤖 Gemini Reply
Promise.race([generateReply(msg, faq, history, handoffMsg), timeout(10s)]) · timeout → GEMINI_UNAVAILABLE
คำอื่นที่เคยเป็น handoff trigger (ขายส่ง/ใบเสนอราคา/ตัวแทนจำหน่าย/เพิ่มกลุ่ม ฯลฯ) จะไม่ถูกดักตรงนี้แล้ว — ปล่อยให้ Gemini ค้น FAQ + search_spender_specs ตามปกติก่อนเสมอ
11
Handle Gemini Returns (ลำดับ)
GEMINI_UNAVAILABLE → ans(UNAVAILABLE_MSG "ระบบนานกว่าปกติ...") → return
CANCEL_IMEI → ans("ยกเลิกรายการแล้ว") + saveHistory([]) → return
HANDOFF / HANDOFF:xxx → SET routed+takeover + DEL notified → notifyAdmin(summary) → ans(handoffMsg) → return
[OUT_OF_DOMAIN] → INCR nonsense_count:{userId} EX 600 · ≤1: ans(OUT_OF_DOMAIN_MSG) · ≥2: เงียบ → return
[NOT_FOUND] → ค้นชีต+เว็บไม่เจอจริง ตรวจ shouldHandoffDeferred ก่อนเสมอ:
มี deferred trigger (ขายส่ง/ใบเสนอราคา/ตัวแทนจำหน่าย/เพิ่มกลุ่ม ฯลฯ): SET pre_handoff:{userId} EX 600 → ans("กรุณาแจ้งรายละเอียด...") → saveHistory → return (ข้าม retry 2 รอบ)
ไม่มี trigger: GET retry:{userId}: ไม่มี = SET EX 600 + ans(retryMsg) + lpush unanswered_log / มี = DEL + ans(handoffMsg) → return
12
Duplicate Answer Suppression
GET last_answer:{userId} · ซ้ำ: INCR repeat_count:{userId} EX 120s · count=1: ans(reminder) · count≥2: เงียบ → return
13
Send Reply — ans()
greetFirst=true: safeReply([offHours?, WELCOME_MSG]) → safePush([answer]) · ส่งแยก 2 ข้อความ
greetFirst=false: safeReply([offHours?, answer])
ต่อจากนั้น: SET last_answer:{userId} EX 120s + DEL repeat_count · saveHistory · log · zincrby question_freq

CORS + Parse Body

Honeypot _hp

Session Blocked

Session Rate Limit

6
Nonsense Filter

Schedule Check
isScheduledOff('web') → "นอกเวลาทำการ ... inbox Facebook ได้เลย" +CONTACT_SUFFIX ครั้งแรก

Load History + Log + FAQ

12
Handle Gemini Returns
GEMINI_UNAVAILABLE → FALLBACK_MSG

≤1 → reply OOD msg +CONTACT_SUFFIX ครั้งแรก

🧠 Gemini 2-Call Flow

ไฟล์: lib/gemini.ts · Model: gemini-2.5-flash · Temperature: 1.0 (text) / 0 (OCR) · Max tokens: 1024
Call 1 — Gemini + SEARCH_TOOL function declaration search_spender_specs(query: string) contents = [...history.map(role+text), {role:'user', text:userMessage}] config: systemInstruction=buildSystemPrompt, tools=[SEARCH_TOOL], temp=1.0, maxTokens=1024
↓ ตรวจ response parts
ไม่มี functionCall
Gemini ตอบจาก FAQ/prompt โดยตรง
finishReason=MAX_TOKENS → DEFAULT_REPLY (=[NOT_FOUND])
→ return response.text.trim()
functionCall: search_spender_specs
query = fcPart.args.query || userMessage
isRecommendQuestion(userMessage)?
YES → searchSpenderRecommend(query) 4 sites
NO → searchSpenderSites(query) 1 combined
↓ ถ้ามี function call
Call 2 — ส่ง functionResponse กลับ Gemini contents = [..., {role:'model', fcPart}, {role:'user', functionResponse:{name, response:{content, url}}}] search ล้มเหลว → content='ไม่พบข้อมูลจากการค้นหา' finishReason=MAX_TOKENS → DEFAULT_REPLY (=[NOT_FOUND]) → return finalResponse.text.trim()
(แก้บั๊กแล้ว) <search_tool> template "ค้นเว็บแล้วยังไม่เจอข้อมูล" ใน prompts.ts เดิม hardcode ประโยค "รอแอดมินติดต่อกลับในเวลาทำการ 08:00–17:00 น." ตายตัวทุกช่องทาง (Gemini ใช้ free text นี้เวลาค้นเว็บเจอข้อมูลใกล้เคียงแต่ไม่ตรงเป๊ะ ซึ่งไม่ผ่าน sentinel [NOT_FOUND] เลย ทำให้ FALLBACK_MSG unification ของ Web Chat ดักไม่ได้) ตอนนี้เปลี่ยนเป็น ${handoffMsg} แล้ว (เหมือนจุดอื่นในไฟล์เดียวกัน) — Web Chat จะได้ FALLBACK_MSG ที่ถูกต้อง (เบอร์ติดต่อ @spenderclub) แทน "รอแอดมิน...เวลาทำการ" ของ LINE/FB
(แก้บั๊กแล้ว) generateReply() เดิมมี API_ERROR_REPLY (ข้อความ hardcode "น้องใจดีไม่พบข้อมูล ต้องการติดต่อแอดมินแจ้งได้เลยนะคะ") return ตรงๆ ตอน finishReason=MAX_TOKENS หรือข้อความว่างหลัง Call 2 — เป็น literal string ไม่ใช่ sentinel ทำให้ web-chat/route.ts (และ LINE/FB) ดักไม่ได้เลย เปลี่ยนเป็น DEFAULT_REPLY ([NOT_FOUND]) ทั้งหมดแล้ว ให้ไหลผ่าน NOT_FOUND handling ปกติของแต่ละช่องทาง · API_ERROR_REPLY ยังใช้อยู่เฉพาะใน generateReplyWithImage() (ไม่แตะ ตามจุดห้ามแตะเรื่องรูปภาพ)

generateReplyWithImage (3 steps)

Step 1 — OCR Extract · maxTokens=400 · temp=0 JSON: {brand, model, code, ocrText} imageQuery = brand + model + code joined · ocrText = ข้อความทุกตัวในรูป
↓
Step 2 — Web Search query = userQuestion || imageQuery → searchSpenderSites(query) → webContext = "ข้อมูลจาก spenderclub.com:\n{text}\nลิงก์: {url}"
↓

Refinement Detection (step 2.5 ใน reasoning_protocol)

ลูกค้าส่งข้อความสั้น เช่น "ใช้ในร้านอาหาร" หลังบอทแนะนำรุ่นไปแล้ว
↓ Gemini อ่าน prompt step 2.5 ก่อน step 3–6
history มีสินค้าที่เคยแนะนำแล้ว
filter รุ่นที่ตรงเงื่อนไขใหม่ → ตอบจาก history ห้าม search ซ้ำ
ถ้าข้อความลูกค้าไม่มีเงื่อนไขใหม่ให้คัดกรองจริง (แค่ย้ำของเดิม เช่น เคยบอก "ร้านอาหาร" แล้วพิมพ์ "ใช้ร้านอาหาร" ซ้ำ) → ห้ามตอบว่าไม่พบข้อมูล ให้ยืนยันรุ่นเดิมที่เคยแนะนำไว้แล้วซ้ำตาม template: "ตามที่แจ้งไปค่ะคุณลูกค้า รุ่น {model_code} เหมาะกับ{เงื่อนไข}ที่ค่ะ 📻 หากลูกค้าต้องการคำแนะนำเพิ่มเติมสามารถ @spenderclub โทร. 085-222-9111, 089-661-2682 ได้ตั้งแต่เวลา 08:00-17:00 น. ค่ะ"
history ไม่มีสินค้าที่เคยแนะนำเลย
กลับไปทำ <recommend_protocol> ใหม่ตั้งแต่ชั้นที่ 1 ทันที (รวมเงื่อนไขนี้เป็นความต้องการ) ห้ามตอบ default_reply เองโดยไม่ค้นก่อน
(แก้บั๊กแล้ว) เดิม step 2.5 ไม่มีกฎรองรับกรณี "history ไม่มีสินค้าที่เคยแนะนำเลย" (เช่น ข้อความก่อนหน้าเป็นแค่คำถามถามลักษณะการใช้งานจากชั้น 1+2 ที่หาไม่เจอ ยังไม่เคยแนะนำสินค้าจริง) — Gemini เจอคำตอบตัวอย่างในกฎ ("ใช้ในร้านอาหาร") ตรงกับข้อความลูกค้าเป๊ะ เข้าใจผิดว่าต้อง filter จาก history ที่ว่างเปล่า แล้วตอบ NOT_FOUND ไปเลยโดยไม่เรียก search_spender_specs เพิ่มกฎ (2) ให้ชัดแล้วว่าต้องกลับไปทำ recommend_protocol ใหม่ในกรณีนี้
(แก้บั๊กเพิ่ม) แม้ history จะมีสินค้าที่เคยแนะนำแล้ว (เข้ากรณี (1)) แต่ถ้าลูกค้าพิมพ์ย้ำเงื่อนไขเดิมโดยไม่มีอะไรใหม่ให้คัดกรอง (เช่น "ใช้สำหรับร้านอาหาร" ซ้ำหลังจากที่บอทแนะนำรุ่นสำหรับร้านอาหารไปแล้ว) Gemini เคยตีความว่า "ไม่มีเงื่อนไขให้ filter" แล้วตอบ NOT_FOUND ทั้งที่ควรแค่ยืนยันคำแนะนำเดิม เพิ่มกฎย่อยใน (1) ให้ชัดแล้ว
